
## Проект

**B2B веб-приложение для автоматизации участия компаний в тендерах и закупках**

## 1. Цель фронтенд-разработки

Разработать современный, быстрый, безопасный и расширяемый фронтенд для B2B веб-приложения, которое помогает компаниям:

* создавать и вести профиль компании;
* загружать документы компании;
* создавать закупки;
* загружать документацию закупки;
* получать AI-разбор закупки;
* просматривать ключевые поля и риски;
* генерировать черновик заявки;
* видеть чек-лист комплектности;
* экспортировать результат;
* пользоваться встроенной справкой и обучающими материалами.

Фронтенд должен быть:

* быстрым;
* простым для пользователя;
* простым в поддержке;
* безопасным;
* адаптивным;
* удобным для поэтапного расширения;
* пригодным для передачи другому владельцу проекта без полной переработки.

# 3. Техническая архитектура фронтенда

## 3.1. Выбранный архитектурный подход

Для MVP должен использоваться подход:

**модульный монолит фронтенда на Next.js**, без микрофронтендов.

Это означает:

* один основной frontend-проект;
* единая дизайн-система;
* единая маршрутизация;
* единые правила авторизации и доступа;
* единые стандарты состояния, логирования, тестирования и UI.

## 3.2. Почему не микрофронтенды

На этапе MVP микрофронтенды не рекомендуются, потому что они:

* усложняют сборку и поддержку;
* увеличивают стоимость разработки;
* усложняют дизайн-систему;
* создают лишние риски рассинхронизации;
* затрудняют передачу проекта новому владельцу.

Для данного проекта оптимальна архитектура, при которой сложность переносится не в количество приложений, а в **грамотное разбиение по внутренним фронтенд-модулям**.

## 3.3. Архитектурная модель рендеринга

Для фронтенда должна использоваться смешанная модель:

* **SSR / Server Components** — для страниц, где важны SEO, быстрый первый рендер и серверная защита;
* **CSR / Client Components** — для интерактивных экранов кабинета, фильтров, таблиц, загрузок и сложных UI-сценариев;
* **динамический импорт** — для тяжелых виджетов и редко используемых частей.

## 3.4. Масштабируемость и отказоустойчивость фронтенда

Фронтенд должен быть собран так, чтобы его можно было расширять без ломки уже работающих страниц.

Обязательные принципы:

* модульная структура;
* разделение UI, бизнес-логики и API-слоя;
* единая схема обработки ошибок;
* повторно используемые компоненты;
* единый слой клиентских запросов;
* централизованные правила авторизации и guard-механизмы;
* возможность переключения источников API и CDN через конфигурацию.

## 3.5. Стратегия интеграции со сторонними API

Фронтенд не должен напрямую обращаться к внешним критичным сервисам, если это затрагивает безопасность, токены, биллинг или AI.

Правила:

* фронтенд работает через backend API;
* прямые вызовы OpenAI, биллинга, внутренних административных сервисов и защищенных файловых endpoint’ов из браузера запрещены;
* для публичных данных допускаются безопасные read-only endpoint’ы;
* фронтенд должен быть готов к замене backend endpoint’ов через конфигурацию без переписывания UI.

## 3.6. Рекомендуемая структура репозитория фронтенда

Рекомендуемая структура:

* `app/` — маршруты и страницы;
* `widgets/` — крупные UI-блоки экранов;
* `features/` — пользовательские действия и сценарии;
* `entities/` — сущности домена;
* `shared/` — общие компоненты, утилиты, типы, хелперы;
* `services/` — клиентский API-слой;
* `hooks/` — общие React hooks;
* `providers/` — глобальные провайдеры;
* `styles/` — глобальные стили и токены;
* `config/` — конфигурация среды;
* `tests/` — тестовые сценарии;
* `types/` — общие типы;
* `public/` — статические файлы.

## 3.7. Клиентское логирование и аналитика ошибок

Во фронтенде должно быть реализовано централизованное логирование клиентских ошибок и событий.

Необходимо предусмотреть:

* централизованный error boundary;
* глобальную обработку runtime-ошибок;
* логирование неуспешных клиентских запросов;
* логирование UI-crash событий;
* сбор технических метрик по ключевым ошибкам;
* безопасную отправку ошибок в серверный лог / monitoring endpoint;
* исключение попадания секретов и чувствительных данных в клиентские логи.

# 4. Безопасность и производительность

## 4.1. Меры безопасности

Во фронтенде необходимо реализовать:

* безопасную аутентификацию через серверную сессионную модель;
* серверную авторизацию по ролям и tenant scope;
* защиту от XSS;
* защиту от CSRF;
* CSP;
* безопасные security headers;
* строгую серверную проверку прав доступа;
* запрет на доверие данным о правах, пришедшим только с клиента.

## 4.2. Хранение секретов и токенов

Во фронтенде запрещено хранить в Local Storage:

* access token;
* refresh token;
* API keys;
* секреты интеграций;
* чувствительные внутренние идентификаторы;
* иные аутентификационные секреты.

Также запрещено использовать Local Storage как постоянный storage для защищенной сессии.

Разрешенная модель:

* HttpOnly cookies;
* Secure cookies;
* SameSite policy;
* серверная сессия;
* краткоживущие клиентские данные только там, где это безопасно и не содержит секретов.

## 4.3. Защита от XSS

Необходимо предусмотреть:

* автоматическое экранирование пользовательских данных;
* запрет небезопасного рендеринга HTML без санитизации;
* использование безопасных компонентов вывода;
* контроль загрузки HTML/markdown-контента из справочных материалов;
* санитизацию пользовательского контента перед выводом.

## 4.4. Защита от CSRF

Если авторизация cookie-based, должны быть предусмотрены:

* CSRF-защита изменяющих запросов;
* серверная проверка происхождения запроса;
* безопасные cookie-флаги;
* разделение read и write запросов по политике доступа.

## 4.5. CSP и security headers

Необходимо внедрить:

* Content-Security-Policy;
* X-Frame-Options / frame-ancestors;
* X-Content-Type-Options;
* Referrer-Policy;
* Permissions-Policy;
* HSTS на уровне прод-инфраструктуры.

## 4.6. Стратегии оптимизации производительности

Во фронтенде должны быть предусмотрены:

* code splitting;
* lazy loading;
* dynamic import для тяжелых модулей;
* оптимизация изображений;
* оптимизация bundle size;
* ограничение количества клиентских библиотек;
* разделение критичного и некритичного JS;
* skeleton loading и плавные состояния ожидания;
* серверный рендер для первых экранов там, где это полезно;
* минимизация лишних повторных запросов.

## 4.7. Работа с CDN и переносимость

Использование CDN должно быть реализовано так, чтобы в будущем можно было безболезненно перейти:

* с зарубежной схемы доставки статики;
* на российский CDN;
* либо на локальную S3-совместимую доставку.

Фронтенд не должен зависеть от одного иностранного CDN настолько, чтобы перенос на российский контур требовал переделки UI.

## 4.8. Клиентское кэширование данных

Кэширование API-данных должно быть реализовано централизованно.

Рекомендуемый подход:

* единый data-fetching слой;
* TanStack Query / React Query или эквивалент;
* четкие cache keys;
* инвалидация после изменений;
* tenant-aware cache keys;
* company-aware cache keys.

Запрещено:

* кэшировать чувствительные данные в Local Storage;
* смешивать данные разных компаний в одном клиентском кэше;
* использовать браузерное постоянное хранилище как источник истины.

## 4.9. Подготовка к нагрузочному тестированию

Фронтенд должен быть подготовлен к нагрузочному и интеграционному тестированию.

Необходимо предусмотреть:

* сценарии авторизации;
* сценарии создания закупки;
* сценарии загрузки файлов;
* сценарии просмотра карточки закупки;
* сценарии запуска генерации;
* сценарии экспорта;
* измерение времени первого рендера;
* измерение взаимодействия с тяжелыми таблицами и списками;
* контроль клиентских ошибок под нагрузкой;
* возможность тестирования с реальным API и mock API.

# 5. UX/UI стандарты

## 5.1. Общие принципы UI

Интерфейс должен быть:

* простым;
* чистым;
* деловым;
* понятным без долгого обучения;
* единообразным;
* адаптивным;
* предсказуемым.

Стиль интерфейса должен соответствовать B2B-продукту, а не маркетплейсу или развлекательному сервису.

## 5.2. Адаптивность и кроссбраузерность

Фронтенд должен корректно работать:

* на desktop;
* на laptop;
* на tablet;
* на mobile в базовом поддерживаемом виде.

Обязательные breakpoints должны быть предусмотрены в дизайн-системе.

Нужно обеспечить корректную работу в актуальных версиях:

* Chrome;
* Edge;
* Safari;
* Firefox.

## 5.3. Доступность

Необходимо учитывать принципы WCAG на базовом практическом уровне.

Нужно предусмотреть:

* достаточный контраст;
* доступность форм и кнопок с клавиатуры;
* правильные label и aria-атрибуты;
* логичный tab order;
* состояния focus;
* понятные сообщения об ошибках;
* доступность модальных окон и dropdown-компонентов.

## 5.4. Дизайн-система и UI-библиотека

Фронтенд должен использовать единую дизайн-систему.

Основа:

* единые токены цвета;
* единые токены отступов;
* единые размеры шрифтов;
* единые правила состояний;
* единые паттерны форм;
* единые таблицы, карточки, бейджи, модальные окна, алерты, пустые состояния, тосты.

Рекомендуется использовать:

* Tailwind CSS;
* shadcn/ui как базу компонентов;
* собственный слой дизайн-правил поверх библиотеки.

## 5.5. Лучшие практики обработки пользовательского ввода

Во фронтенде необходимо реализовать:

* понятную валидацию форм;
* разделение клиентской и серверной валидации;
* нормальные тексты ошибок;
* маски ввода там, где это уместно;
* предупреждение до потери несохраненных данных;
* защиту от случайной двойной отправки формы;
* понятную обратную связь при загрузке и сохранении;
* отдельные состояния:

  * loading;
  * success;
  * validation error;
  * server error;
  * empty state.

## 5.6. UX для сложных экранов

Для экранов закупок, чек-листов, документов и AI-разбора нужно предусмотреть:

* удобную табличную навигацию;
* фильтры;
* сортировку;
* статусные бейджи;
* явное выделение рисков;
* блок “нужна проверка”;
* понятную иерархию важности полей;
* фиксированную верхнюю область с ключевыми данными;
* понятную визуальную связь между найденным полем и исходным фрагментом текста.

# 6. Оптимизация процессов разработки и ИИ

## 6.1. Стандарты тестирования

Во фронтенд-проекте должны быть унифицированные стандарты тестирования.

Необходимо предусмотреть:

* unit tests для утилит, функций преобразования, валидаторов и бизнес-логики фронтенда;
* integration tests для форм, клиентских сценариев, таблиц, фильтров и экранов;
* E2E tests для ключевых пользовательских потоков;
* smoke tests после деплоя;
* отдельные тесты для авторизации, загрузки файлов, создания закупки, AI-разбора, генерации и экспорта.

## 6.2. CI/CD для фронтенда

Необходимо спроектировать CI/CD pipeline, который включает:

* установку зависимостей;
* type check;
* lint;
* unit tests;
* integration tests;
* build;
* проверку env-конфигурации;
* optional preview environment для review;
* деплой в stage;
* ручной или контролируемый деплой в production.

В pipeline также должны быть предусмотрены:

* запрет на деплой при провале тестов;
* проверка bundle size на регрессии;
* проверка security и dependency audit;
* возможность раздельного деплоя stage и production.

## 6.3. Код-стайл, линтинг и форматирование

Во фронтенде должны быть единые правила:

* TypeScript strict mode;
* ESLint;
* Prettier;
* единые naming conventions;
* единый стиль именования файлов, компонентов, hooks, DTO и API-функций;
* запрет “грязных” импортов и циклических зависимостей;
* запрет крупных универсальных файлов на сотни строк без декомпозиции.

## 6.4. Управление зависимостями

Необходимо минимизировать хаотичный рост зависимостей.

Требования:

* каждая новая библиотека должна быть обоснована;
* не допускается подключение нескольких библиотек с одинаковой ролью без причины;
* зависимости должны обновляться по понятной стратегии;
* должны быть правила для безопасного обновления major-версий;
* не должно быть критической зависимости от библиотеки, которую сложно заменить при переносе проекта другому владельцу.

## 6.5. Роль ИИ в разработке

ИИ должен использоваться как инструмент ускорения разработки, но не как замена архитектурного контроля.

ИИ можно использовать для:

* генерации типовых компонентов;
* генерации тестов;
* подготовки refactor-предложений;
* проверки на дублирование кода;
* поиска слабых мест в доступности и UI;
* первичной генерации документации;
* ускорения code review.

ИИ не должен:

* самостоятельно определять архитектурные решения без проверки;
* добавлять зависимости без оценки;
* генерировать небезопасный код без ручной проверки;
* менять контракты API без согласования.

## 6.6. Процесс code review

Нужно сформировать процесс ревью, в котором проверяются:

* архитектурная чистота;
* типобезопасность;
* безопасность;
* производительность;
* UX-согласованность;
* соответствие дизайн-системе;
* тестовое покрытие;
* отсутствие утечек секретов;
* отсутствие хранения чувствительных данных в Local Storage;
* отсутствие обхода серверной авторизации.

Допускается использовать ИИ как вспомогательного ревьюера, но финальное решение должно приниматься человеком.

# 10. Критерии приемки фронтенда

Фронтенд считается принятым, если:

* реализованы все ключевые экраны MVP;
* работает регистрация, вход и защищенная зона;
* права доступа проверяются корректно;
* секреты и токены не хранятся в Local Storage;
* формы имеют валидацию и понятную обратную связь;
* интерфейс адаптивен;
* UI единообразен;
* предусмотрены пустые состояния, ошибки и loading-состояния;
* данные компании не смешиваются между разными tenant / company;
* AI-разбор отображается понятно и структурированно;
* фронтенд готов к подключению и работе с российским контуром без переписывания логики;
* клиентское логирование и error handling реализованы;
* настроены тесты, линтинг и build pipeline;
* реализованы меры защиты от XSS, CSRF и применен CSP;
* производительность не деградирует на типовых сценариях;
* проект структурирован так, чтобы новые задачи минимально затрагивали старые участки кода.

# 11. Итоговое архитектурное решение для фронтенда

Для данного проекта фронтенд должен строиться как:

* **единое Next.js-приложение**;
* **модульный монолит**;
* **с четким разделением на app / features / entities / shared**;
* **с серверной моделью безопасности**;
* **с безопасной cookie-based авторизацией**;
* **без хранения секретов в Local Storage**;
* **с централизованным клиентским API-слоем**;
* **с переносимой конфигурацией под зарубежный и российский контур**;
* **с возможностью поэтапного роста без перехода на микрофронтенды в MVP**.
