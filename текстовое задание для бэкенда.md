1. Назначение бэкенда
Бэкенд должен обеспечивать основной продуктовый контур системы: регистрацию и авторизацию, управление компаниями и профилями компаний, хранение документов компании, работу с закупками и файлами закупки, извлечение текста, AI-разбор через отдельный AI-gateway, генерацию черновика заявки, чек-лист комплектности, экспорт, справочный раздел, тарифные ограничения, аудит, фоновые задачи и административный контроль.
Бэкенд должен быть безопасным, переносимым между зарубежным и российским контуром, понятным для передачи другому владельцу и пригодным к дальнейшему выделению модулей в отдельные сервисы.
2. Архитектурная модель
На старте система реализуется как модульный монолит с отдельным worker-процессом для фоновых задач и отдельным внешним AI-gateway.
Внутри бэкенда должны быть разделены доменная логика, application layer, контракты модулей, инфраструктурные адаптеры, очереди и интеграционный слой.
2.1. Разделение контуров
•	Основной продуктовый контур: веб-приложение, backend, PostgreSQL, Redis, объектное хранилище, фоновые воркеры, аудит, биллинг, справочный раздел, админка.
•	Внешний AI-контур: AI-gateway, интеграция с OpenAI API, маршрутизация моделей, минимизация и маскирование передаваемых данных, контроль токенов и стоимости.
2.2. Запрещённые практики
•	Нельзя вызывать внешние сервисы напрямую из UI.
•	Нельзя смешивать доменные правила и SQL/ORM-реализацию в одном слое.
•	Нельзя хранить AI-промпты и AI-логику в UI-компонентах.
•	Нельзя закладывать прямой вызов OpenAI из российского продуктового контура.
2.3. Правило простого переезда
После переноса в российскую инфраструктуру в РФ должны находиться backend, PostgreSQL, Redis, файловое хранилище, фоновые воркеры, админка, аудит, справочный раздел и пользовательские данные. Во внешнем контуре остаются AI-gateway и OpenAI-интеграция. Переезд должен требовать в основном смены env, переноса БД и хранилища, а не переписывания бизнес-логики.
3. Рекомендуемый стек
•	Backend runtime: Next.js server layer.
•	Application layer: отдельный слой use case’ов.
•	Worker: отдельный процесс фоновых задач.
•	База данных: PostgreSQL.
•	ORM: Prisma.
•	Кэш и очереди: Redis.
•	Файловое хранилище: S3-совместимое объектное хранилище.
•	OCR и обработка документов: extraction layer + OCR adapter.
•	AI: отдельный AI-gateway.
•	Инфраструктура: Docker, docker-compose, reverse proxy, резервное копирование БД и файлов.
4. Модульная структура бэкенда
Бэкенд должен быть разделён на модули: auth, users, companies, company_profile, company_documents, knowledge_base, tenders, tender_files, text_extraction, parser, ai_gateway_client, generator, checklist, export, billing, notifications, admin, audit_log, infrastructure_settings.
Каждый модуль обязан иметь свою зону ответственности, свои use case’ы, DTO, сервисы, инфраструктурные адаптеры и тесты.
5. Основные сущности БД
Минимально должны быть предусмотрены сущности: User, Session, Company, CompanyUser, CompanyProfile, CompanyDocument, CompanyDocumentVersion, Tender, TenderFile, TenderExtractionJob, TenderParseResult, TenderField, TenderRequirement, DraftGeneration, ChecklistItem, Subscription, Plan, UsageCounter, AuditLog, Notification, KnowledgeMaterial, KnowledgeMaterialVersion, ProviderConfig, BackgroundJob, IntegrationEvent.
5.1. Правила работы со схемой
•	Все изменения схемы БД только через migration scripts.
•	Все сущности, завязанные на tenant/company scope, должны явно хранить ссылку на Company.
•	AI-операции, фоновые задачи и экспортные форматы должны быть версионируемыми.
6. Пользователи, роли и доступ
Минимальные роли: Гость, Пользователь компании, Администратор компании, Системный администратор.
•	Пользователь компании: создаёт компанию, заполняет профиль, загружает документы компании, создаёт закупки, загружает документацию закупки, запускает AI-разбор, просматривает результаты, запускает генерацию при наличии тарифа, экспортирует результат, пользуется справочным разделом.
•	Администратор компании: управляет данными компании, видит тариф и лимиты, управляет доступами внутри компании в базовом объёме, видит историю действий своей компании.
•	Системный администратор: видит пользователей и компании, управляет тарифами, очередями, ошибками, повторной обработкой, злоупотреблениями, справочными материалами, AI-интеграцией и feature flags.
Проверка tenant scope и company scope должна происходить на сервере, а не только в UI.
7. Auth и сессионная модель
•	Регистрация по email и паролю.
•	Подтверждение email.
•	Вход и выход.
•	Восстановление пароля.
•	Безопасное хранение аутентификационных данных.
•	Управление сессиями.
•	Защита от brute force.
•	Логирование попыток входа.
Секреты, токены, API-ключи и иные чувствительные данные запрещено хранить в Local Storage и Session Storage. Используется серверная сессионная модель с HttpOnly и Secure cookies.
8. Компании и профиль компании
Бэкенд должен позволять создавать и редактировать компанию, хранить профиль компании, реквизиты, сведения о руководителе, адреса, контактное лицо, email, телефон, банковские реквизиты и дополнительные сведения для шаблонов заявки.
8.1. Критичное правило для AI
Полный профиль компании не должен по умолчанию уходить во внешний AI-контур. Данные профиля используются внутри основного контура для локального структурированного шаблона черновика, локальной подстановки реквизитов и локального сопоставления с требованиями закупки. Во внешний AI-контур допускается передавать только отдельные нормализованные и минимальные фрагменты, если без этого нельзя выполнить конкретную AI-задачу.
9. Документы компании
Бэкенд должен обеспечивать загрузку документов компании, типизацию, срок действия, статусы, версионность, историю изменений, приватное хранение, скачивание и удаление по правам, а также привязку документа к Company и сопоставление документа с чек-листом закупки.
10. Закупки и документация закупки
•	Создание закупки вручную, по ссылке и через загрузку файлов.
•	Хранение списка файлов закупки и их статусов обработки.
•	Поддерживаемые форматы: PDF, DOCX, DOC, XLSX, JPG, PNG, ZIP.
•	PDF: извлечение текстового слоя, иначе OCR fallback.
•	ZIP: безопасная распаковка во временную карантинную зону.
10.1. Ограничения безопасности для ZIP
•	Лимит максимального размера архива.
•	Лимит числа файлов внутри.
•	Лимит глубины вложенности.
•	Запрет опасных расширений.
•	Защита от zip bomb.
•	Временное карантинное хранение до завершения проверки.
11. AI-разбор закупки
Система должна выделять предмет закупки, заказчика, номер закупки, НМЦК, валюту, даты, срок и место поставки, обеспечения, требования к участнику, состав заявки, обязательные документы, гарантии, риски и спорные места.
Ключевые поля НМЦК, срок поставки и место поставки должны быть вынесены отдельно. Если AI не уверен, поле помечается как требующее проверки, а пользователю показывается исходный фрагмент.
AI-разбор вызывается не напрямую из бизнес-модуля, а только через отдельный AI-gateway, который принимает нормализованный вход, выбирает шаблон разбора, вызывает модель и возвращает структурированный ответ.
12. AI-интеграция, минимизация и персональные данные
12.1. Базовый принцип
AI-интеграция должна быть построена по принципам privacy by design и data minimization. Во внешний AI-контур передаётся только минимально необходимый объём данных, достаточный для конкретной операции.
12.2. Что обязательно делать перед AI-вызовом
•	Локально извлекать текст.
•	Разбивать текст на фрагменты.
•	Находить релевантные участки.
•	Выделять вероятные блоки с НМЦК, сроками, местом поставки, требованиями и составом заявки.
•	Передавать наружу только релевантные фрагменты, нормализованные выдержки, поля-кандидаты и короткие контекстные блоки.
12.3. Что запрещено отправлять по умолчанию
•	Полный профиль компании.
•	Полный комплект документов компании.
•	Весь архив закупки целиком.
•	Лишние контактные данные и банковские реквизиты.
•	Полные доверенности, паспортные данные и внутренние документы компании.
•	Историю действий пользователя и иные не относящиеся к операции данные.
12.4. Маскирование и суррогатные идентификаторы
Если AI-задача не требует точной идентификации лица или компании, backend должен по возможности маскировать ФИО, телефоны, email, паспортные данные, доверенности и реквизиты, заменять внутренние идентификаторы на surrogate-id и вырезать лишние персональные и служебные данные.
12.5. Обязательные механизмы AI-приватности
•	Маскирование чувствительных полей.
•	Выборочный фрагментарный экспорт текста.
•	Предикат «можно / нельзя отправлять во внешний AI-контур».
•	Журнал причин отправки.
•	Настройки уровня приватности для AI-обработки.
•	Возможность принудительно отключить AI для отдельных компаний или типов документов.
12.6. Классификация AI-запроса
Каждый AI-запрос должен сопровождаться метаданными: тип операции, уровень чувствительности, признак маскировки, используемая модель, версия промпта, объём токенов, результат внутренней валидации ответа.
12.7. Двухэтапная генерация черновика
Генерация черновика заявки работает в два этапа: сначала внутри основного контура локально формируется структурированный шаблон с подстановкой реквизитов и данных профиля, затем во внешний AI-контур при необходимости уходят только отдельные части, где нужен языковой или смысловой анализ. Полный профиль компании, полный комплект внутренних документов и лишние персональные данные не должны попадать в AI-промпт по умолчанию.
12.8. Модельная стратегия
•	Основной маршрут MVP — GPT-5 mini.
•	Сверхдешёвые операции — GPT-5 nano или GPT-5.4 nano.
•	Сложные, спорные и low-confidence случаи — эскалация на GPT-5.4 mini и точечно на GPT-5.4.
•	Выбор модели происходит внутри AI-gateway без изменения доменной логики.
12.9. После переезда на российские серверы
Переезд основного контура в РФ не считается достаточной защитой сам по себе. Backend обязан сохранить ту же модель: OpenAI не вызывается напрямую из российского контура, используется внешний AI-gateway, между контурами идёт защищённый сервисный канал, а во внешний AI-контур уходят только минимально необходимые данные, при необходимости маскированные и урезанные.
13. Генерация черновика заявки
Бэкенд должен формировать черновой комплект заявки, подставлять реквизиты компании, использовать данные профиля компании, определять список необходимых приложений и отмечать поля, требующие ручной проверки. Генерация не должна позиционироваться как полностью юридически гарантированная.
14. Сопоставление с документами компании
Система должна сравнивать требования закупки с документами компании, определять, какие документы есть, каких не хватает и какие истекли, используя результаты в чек-листе комплектности.
15. Чек-лист комплектности
Чек-лист должен формироваться по закупке и содержать статусы: готово, не найдено, найдено, но истёк срок, нужна ручная проверка.
16. Экспорт
Бэкенд должен обеспечивать экспорт карточки разбора закупки, чек-листа, чернового комплекта заявки и комплекта файлов для скачивания. Форматы MVP: PDF, DOCX, ZIP с комплектом файлов.
17. Очереди и фоновые задачи
•	Очередь и отдельный worker для обработки загрузки файлов, извлечения текста, OCR fallback, безопасной обработки ZIP, AI-разбора, генерации черновика, построения чек-листа, экспорта и повторной обработки после ошибки.
•	Каждая задача должна иметь тип, статус, инициатора, tenant/company scope, payload, retry policy, timestamps, причину ошибки и связь с продуктовой сущностью.
•	Минимальные статусы задач: pending, queued, processing, done, failed, retry_scheduled, canceled.
18. Knowledge base и справочный раздел
Бэкенд должен поддерживать хранение справочных материалов, версии, категории, привязку к экрану или разделу, актуальность и архивность, загрузку и замену файлов, выдачу пользователям актуальной версии и админское управление материалами.
19. Админ-панель
Админ-бэкенд должен позволять видеть пользователей и компании, управлять тарифами, лимитами, очередями задач, ошибками, повторной обработкой, злоупотреблениями, справочными материалами, конфигурацией провайдеров, состоянием AI-gateway, техническими журналами фоновых задач и feature flags.
20. API и серверные ограничения
•	Все приватные endpoint’ы должны проверять серверную сессию, роль, tenant scope и company scope.
•	Все endpoint’ы валидируют входные данные и защищают внутренние маршруты.
•	Обязательны rate limiting и idempotency для повторяемых операций, где это требуется.
•	Необходимо логировать авторизацию, загрузку, удаление, смену тарифа, запуск AI-операций, ошибки, создание закупки, запуск генерации и экспорт.
21. Безопасность файлов
Бэкенд обязан обеспечивать приватное хранение файлов, доступ только авторизованным пользователям, проверку MIME type, ограничение размера файлов, проверку на недопустимые расширения, карантин до завершения проверки и безопасную распаковку ZIP.
22. Секреты и конфигурация
•	Хранение секретов вне кода.
•	Раздельные ключи для dev, stage и prod.
•	Отдельные ключи для AI-gateway.
•	Отдельные сервисные учётные записи.
•	Ротация ключей и журнал изменения критической конфигурации.
23. Нефункциональные требования
•	Авторизация — до 2 секунд.
•	Создание закупки — до 2 секунд.
•	Старт обработки после загрузки файла — сразу после завершения загрузки.
•	Разбор типовой закупки — до 30–90 секунд.
•	Генерация черновика — до 60 секунд.
•	Архитектура должна быть подготовлена к централизованным логам, метрикам, health-check endpoints и алертам.
24. Этапы backend-разработки
Этап	Содержание
1	Каркас проекта, авторизация, компания, БД, роли, изоляция данных, файловое хранилище, базовая документация.
2	Документы компании, типизация, сроки действия, статусы документов, история версий, справочный раздел MVP.
3	Создание закупки, загрузка файлов, карточка закупки, очереди обработки, extraction layer.
4	Извлечение текста из PDF, DOCX, DOC, XLSX, JPG, PNG, ZIP; OCR fallback; карантин и безопасная проверка файлов.
5	AI-разбор, карточка закупки, AI-gateway, маршрутизация моделей, маскирование персональных данных.
6	Генерация черновика, сопоставление с документами компании, чек-лист, экспорт.
7	Тарифы, демо-лимит, Стартер, биллинг, ограничения функций, счётчики использования.
8	Админка, аудит, повторная обработка, стабилизация, пакет передачи проекта, миграционный сценарий на российский сервер.
9	Тарифы Анализ, Плюс, затем Про и Бизнес.
25. Критерии приёмки backend MVP
•	Пользователь может зарегистрироваться и войти.
•	Пользователь может создать компанию и загрузить документы компании.
•	Пользователь может создать закупку и загрузить документы закупки.
•	Система извлекает текст из поддерживаемых форматов.
•	Система выделяет НМЦК, срок поставки, место поставки, требования к участнику, состав заявки и риски.
•	Пользователь видит карточку закупки и при наличии тарифа получает генерацию черновика.
•	Система строит чек-лист комплектности и позволяет экспортировать результат.
•	Данные одной компании изолированы от другой.
•	AI вынесен в отдельный AI-gateway, а наружу передаются только минимально необходимые данные.
•	Персональные данные перед внешним AI-вызовом маскируются и урезаются по правилам системы.
26. Пакет передачи проекта
К передаче должны входить исходный код, схема БД, ER-диаграмма, диаграмма модулей, схема размещения контуров, перечень env, инструкции по развертыванию, миграции, резервному копированию и восстановлению, список внешних сервисов, описание фоновых задач, описание API-модулей, тарифных правил и runbook поддержки.
27. Жёсткие backend-условия, которые нельзя нарушать
•	Нельзя отправлять полный профиль компании во внешний AI-контур без явной необходимости.
•	Нельзя отправлять полный комплект документов компании во внешний AI-контур по умолчанию.
•	Нельзя строить генерацию черновика как прямую схему «полная заявка целиком в OpenAI».
•	Нельзя вызывать OpenAI напрямую из российского продуктового контура.
•	Нельзя хранить секреты в коде или в клиентском хранилище браузера.
•	Нельзя выпускать API без rate limiting, валидации ролей и company scope.
•	Нельзя принимать ZIP без ограничений размера, числа файлов и глубины вложенности.
•	Нельзя смешивать AI-промпты, бизнес-логику и инфраструктурный код в одном слое.
•	Нельзя добавлять новые backend-функции без миграций, use case’ов и контракта модуля.
•	Нельзя считать переезд в РФ достаточной мерой сам по себе: минимизация и маскирование перед AI остаются обязательными.
 
Приложение А. Прикладная версия для Cursor
Этот раздел нужен для постановки задач агентам и разработчикам. Он описывает, что именно должно появиться по папкам, Prisma-моделям, route handlers, очередям и worker-задачам. Здесь не меняется смысл полного ТЗ, а добавляется практический уровень реализации.
А.1. Рекомендуемая структура каталогов
•	apps/web/app/api/... — route handlers, только тонкий слой HTTP и валидация.
•	apps/web/lib/auth/... — auth, session, company-scope, guards.
•	apps/web/lib/storage/... — S3-адаптеры и безопасная работа с файлами.
•	apps/web/lib/ai/... — клиент AI-gateway и локальные политики минимизации данных.
•	apps/web/lib/use-cases/... — application layer и use case’ы.
•	packages/db/prisma/... — schema.prisma и миграции.
•	packages/contracts/... — DTO, типы ответов, enum и контракты между модулями.
•	packages/core/... — доменные типы, value objects, общие правила.
•	packages/integrations/... — интеграции с AI-gateway, storage и внешними сервисами.
•	apps/worker/src/... — обработчики очередей и фоновые задачи.
А.2. Prisma-модели, которые должны быть в базе
Модель	Назначение	Ключевые связи
Company	Компания-tenant	1:N с CompanyUser, 1:1 с CompanyProfile
CompanyProfile	Расширенный профиль компании	1:1 с Company
CompanyDocument	Документ компании	1:N с CompanyDocumentVersion
Tender	Карточка закупки	1:N с TenderFile, TenderParseResult, DraftGeneration
TenderFile	Файл закупки	N:1 к Tender
TenderParseResult	Результат AI-разбора	N:1 к Tender
ChecklistItem	Пункт чек-листа	N:1 к Tender
KnowledgeMaterial	Справочный материал	1:N с KnowledgeMaterialVersion
BackgroundJob	Фоновая задача	N:1 к Company и User
А.3. Route handlers, которые должны существовать
•	GET /api/health — liveness.
•	GET /api/health/ready — readiness по PostgreSQL и ключевым зависимостям.
•	POST /api/auth/register, POST /api/auth/login, POST /api/auth/logout, GET /api/auth/me.
•	POST /api/company, PATCH /api/company, GET /api/company/me.
•	GET /api/company/profile, PATCH /api/company/profile.
•	GET /api/company/documents, POST /api/company/documents, POST /api/company/documents/[documentId]/upload.
•	GET /api/tenders, POST /api/tenders, GET /api/tenders/[id].
•	POST /api/tenders/[id]/files — загрузка файлов закупки.
•	POST /api/tenders/[id]/parse — постановка AI-разбора в очередь.
•	POST /api/tenders/[id]/generate-draft — постановка генерации в очередь.
•	GET /api/knowledge/materials, GET /api/knowledge/materials/[id].
•	GET /api/admin/jobs, GET/POST/PATCH /api/admin/knowledge/materials...
А.4. Очереди и worker-задачи
•	tender.fileRegistered — постановка обработки загруженного файла.
•	tender.extractText — извлечение текста и OCR fallback.
•	tender.parse — AI-разбор закупки через AI-gateway.
•	tender.generateDraft — двухэтапная генерация черновика.
•	tender.buildChecklist — формирование чек-листа комплектности.
•	tender.export — подготовка экспортного комплекта.
•	knowledge.reindex — при изменении справочных материалов, если это потребуется.
•	Каждая очередь обязана записывать BackgroundJob и синхронно обновлять статусы processing, done, failed.
А.5. Правило реализации AI-гейтвея
•	Любой вызов модели идёт только через AI-gateway client.
•	Перед AI-вызовом выполняется локальная минимизация, фрагментация и маскирование данных.
•	Должен существовать слой policy, который отвечает на вопрос: можно ли отправлять этот набор данных во внешний AI-контур.
•	Должен существовать слой serializer, который формирует безопасный AI-payload.
•	Должен существовать журнал AI-request meta: тип операции, уровень чувствительности, маскирование, модель, версия промпта, токены, результат валидации.
А.6. Что не должен делать Cursor-агент
•	Не делать git commit и git push без явной команды.
•	Не трогать .pnpm-store и иные локальные кэши.
•	Не перезапускать полный pnpm install без необходимости.
•	Не менять архитектуру модульного монолита без отдельного решения.
•	Не отправлять во внешний AI-контур полный профиль компании и полные документы компании.
•	Не добавлять новые backend-функции без миграции, контракта и use case’а.
А.7. Чек-лист при приёмке каждой backend-задачи
•	Есть миграция Prisma, если менялась схема.
•	Есть серверная проверка роли и company scope.
•	Есть валидация входных данных.
•	Есть аудит ключевых действий.
•	Если есть AI-вызов — есть минимизация и маскирование.
•	Если есть фоновые задачи — есть запись BackgroundJob.
•	Есть инструкция, как проверить функционал локально.
•	Нет автокоммитов, автопушей и случайного мусора в репозитории.
